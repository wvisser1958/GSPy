Yes. What you’re seeing is entirely plausible.

The short version is that the 2000 K limit is real for the phase data, but whether you actually crash depends on which equilibrium solver path is used, what intermediate temperatures it visits, and whether C5H5 is present in the relevant intermediate/equilibrium composition in a numerically important way. Cantera’s solvers are not all the same here. The fast ChemEquil / element-potential solver is a different algorithm from the more robust gibbs and vcs solvers. In current Cantera docs, equilibrate(..., solver='auto') tries one solver and falls back to another if needed; historically and across versions, auto behavior has differed, which is one reason these cases can feel inconsistent.

What matters most is this distinction:

gas.max_temp == 2000 K means the thermo parameterization is only guaranteed valid up to 2000 K for the whole phase, because one species (C5H5) tops out there.
That does not automatically mean every HP equilibrium call must fail whenever the final reported temperature is above 2000 K.

Why not? Because for HP, the robust gibbs and vcs approaches effectively solve a sequence of TP equilibrium problems inside an outer iteration on temperature/pressure. Cantera’s docs explicitly note that for gibbs and vcs, max_iter is the maximum number of outer temperature or pressure iterations when T and/or P is not fixed. By contrast, ChemEquil has an explicit range check that can throw Temperature ... outside valid range when it encounters an out-of-range temperature during its procedure.

So the “inconsistency” is usually one of these:

Different solver got used.
In one run, auto may end up using a more robust route; in another, the fast element-potential path may be attempted and fail when it touches an out-of-range state. Cantera documents distinct solver options (element_potential / ChemEquil, gibbs, vcs, auto) and that auto tries solvers in sequence.
The final state can be above 2000 K even if the numerics never “need” C5H5.
If at the final hot lean equilibrium C5H5 is essentially zero, the solver may converge to a composition dominated by stable small species (CO2, H2O, CO, O2, N2, radicals) and never get hung up badly on C5H5. But in another case, an intermediate iterate or a different starting point can assign enough C5H5 to matter numerically, triggering the temperature-range problem or destabilizing the search. This is an inference from the documented solver differences plus your observation that the issue appears only in some cases.
Sensitivity to starting point.
You already observed this with your larger gas-turbine loop: far-from-solution starting points can make the trimmed mechanism less stable. That is very consistent with Cantera’s HP implementation, where non-TP equilibrium uses outer iterations on temperature/pressure and can fail with the well-known setState_HPorUV “No convergence in 500 iterations” path when the iteration wanders badly.

Here’s how I would make your combustor block more numerically bulletproof.

1. Fix the solver explicitly

Do not rely on auto in a large system loop.

Use:

gas.equilibrate("HP", solver="gibbs", max_iter=1000)

Why:

element_potential is faster but less robust.
gibbs is slower but more robust.
auto can behave differently depending on the case, which is exactly the inconsistency you observed.

If gibbs still struggles in rare cases, try:

gas.equilibrate("HP", solver="vcs", max_iter=2000)

vcs is often even more forgiving, though typically slower. Cantera lists vcs as the non-ideal equilibrium solver and gibbs as the robust Gibbs minimization solver.

2. Never start the combustor equilibrium from 298 K unless you truly mean it

Your combustor inlet is already hot. Starting far away makes the outer HP iteration work much harder.

Bad:

gas.TP = 298.15, P

Better:

gas.TP = T_air_in, P_comb

or even:

gas.TP = max(T_air_in, 800.0), P_comb

for a hot combustor branch.

This matters because HP equilibrium involves outer T/P iterations, and bad starting temperatures are one common reason those iterations wander into ugly regions.

3. Reuse the previous converged combustor state as the next initial guess

This is one of the biggest stability wins in a cycle loop.

Instead of rebuilding the gas object from scratch every time, preserve:

previous temperature
previous composition
previous pressure

Then update only what changed.

Pattern:

# previous converged state from last nonlinear iteration
# gas already contains a realistic hot composition

gas.P = P_comb
gas.HP = h_target, P_comb
gas.equilibrate("HP", solver="gibbs", max_iter=1000)

or, if you must rebuild composition each time, at least initialize with the previous combustor temperature and composition.

This reduces solver travel dramatically.

4. Add a guarded fallback ladder

For a production turbine code, I would not let one failed equilibrium call kill the whole outer loop immediately.

Use a fallback sequence like this:

def robust_hp_equilibrate(gas, h_target, p_target):
    # Try best default
    try:
        gas.HP = h_target, p_target
        gas.equilibrate("HP", solver="gibbs", max_iter=1000)
        return "gibbs"
    except Exception:
        pass

    # Retry with a better temperature guess
    try:
        gas.TP = max(gas.T, 1200.0), p_target
        gas.HP = h_target, p_target
        gas.equilibrate("HP", solver="gibbs", max_iter=2000)
        return "gibbs-retry"
    except Exception:
        pass

    # Final fallback
    gas.TP = max(gas.T, 1200.0), p_target
    gas.HP = h_target, p_target
    gas.equilibrate("HP", solver="vcs", max_iter=2000)
    return "vcs"

That keeps the loop moving and makes failures much rarer.

When you get the setState_HPorUV error, Cantera is telling you the target enthalpy, current pressure, starting temperature, and current temperature because the temperature iteration itself did not converge. That error format comes directly from ThermoPhase::setState_HPorUV.

Log these at failure:

target enthalpy
pressure
current inlet composition
equivalence ratio
starting temperature
fuel flow
oxidizer flow
previous converged state

That makes it much easier to distinguish:

bad initial guess
unreachable state
solver path issue
mechanism issue
6. Add a physically reasonable pre-check before HP equilibrium

Before calling equilibrium, estimate whether the requested state is obviously absurd.

Examples:

combustor pressure must be positive and realistic
fuel/air ratio should be inside a plausible band
inlet enthalpy should not be grossly inconsistent with the species set

A lot of “solver instability” in cycle codes is really edge-case state generation upstream.
